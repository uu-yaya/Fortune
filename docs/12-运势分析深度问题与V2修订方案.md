# 运势分析深度问题与 V2 修订方案（基于 docs/13 二次审查）

## 1. 文档目的
- 汇总本次线上问题的本质分析（不仅限于单个问句）。
- 结合测试文档 `docs/13-运势分析V2问题测试结果-2026-02-27.md` 做二次审查。
- 输出可落地、可灰度、可回滚的修订版方案 V2。
- 给出按优先级拆解的改造步骤与验收标准。

## 2. 本次问题复盘（现象）
- 用户询问不同运势问题时，回复高频出现同一结构：
  - `先给你结论 / 命理信号 / 命理依据 / 五行分布 / 行动建议 / 参考置信度`
- 即便问题语义不同（例如“本周运势” vs “开源优先还是守财优先”），回复骨架和建议风格仍高度相似。
- 用户体感：
  - 形式完整，但分析深度不够；
  - 口吻偏模板化，人设感弱；
  - 对“二选一/比较型问题”的直答不稳定。

### 3.3 实测结论
- 系统“可返回稳定结构”能力正常。
- 系统“按问题类型精确回答”能力不足。
- 关键缺口：
  1. 决策型直答不稳定。
  2. 星座缺信息不澄清。
  3. 趋势/口语时窗不落时间窗口。
  4. 连续追问仍出现高同质模板。
  5. 非 time 分支存在日期/星期/相对时窗偶发错配。

## 5. 二次审查结论（从问题到改造优先级）
- 当前最优改造顺序不是“先美化口吻”，而是：
  1. 先修“答没答到点上”（决策直答、澄清、时窗、时序一致性）。
  2. 再修“同模板重复”和不稳定分支可观测性。
  3. 最后做门禁和灰度固化。
- 若顺序反过来，会出现“口吻更可爱但仍答非所问”的假改善。

## 6. 修订版方案 V2（总览）
- 架构分层：
  1. 意图理解层（domain_intent + question_type）
  2. 结构化决策层（标准化中间对象）
  3. 用户态表达层（吉伊口吻自然表达）
- 发布策略：全链路 feature flag + 灰度 + 自动回滚。
- 质量门禁：从“字段齐全”升级为“问题直答 + 分析深度 + 重复率 + 人设一致 + 稳定性 + 时序一致性 + 可观测性完整度”。

## 7. 分阶段落地（按实测缺口重排）

### P0：开关化与回滚能力（发布安全网）

#### 改造项
- 增加开关：
  - `intent_v2`
  - `clarify_v2`
  - `window_v2`
  - `render_v2`
  - `quality_gate_v2`
- 任一开关关闭时，回退当前稳定链路。
- 新增开关优先级（避免组合歧义）：
  - `intent_v2` -> `clarify_v2/window_v2` -> `render_v2` -> `quality_gate_v2`
- 新增非法组合约束：
  - `quality_gate_v2=on` 且 `render_v2=off` 时，质量判定仅允许基于结构对象，禁止基于文案判定。
  - `window_v2=on` 且 `intent_v2=off` 时，直接降级为旧链路并打日志 `reason_code=window_without_intent`。

#### 测试样例
- 用例 P0-01：`render_v2=off`
  - 输入：`给我算算本周运势`
  - 预期：保持当前线上行为。
- 用例 P0-02：`render_v2=on`
  - 输入：`给我算算本周运势`
  - 预期：进入新链路，日志可区分路径。

#### 通过标准
- 开关切换后 1 分钟内生效。
- 回滚不需要重新发版。
- 关键链路日志覆盖率 >= 99%（必须含 `route_path`、`reason_code`、`flag_snapshot`）。

### P1：先修“答到点上”

#### P1-A 决策型直答
- 改造项：
  - 识别 `question_type=decision/comparison`。
  - 强制首句包含明确策略结论（如“先守后开/守中带开”）。
- 测试样例：
  - 输入：`我最近财运是开源优先还是守财优先？`
  - 输入：`这阵子财运我该先扩收入还是先控支出？`
- 通过标准：
  - 决策型首句直答命中率 >= 95%。

#### P1-B 星座缺信息澄清
- 改造项：
  - 对“星座运势”且未提供星座名，先澄清，不入八字输出。
- 测试样例：
  - 输入：`我的星座运势`
  - 输入：`想看我的星座最近运势`
- 通过标准：
  - 缺信息澄清命中率 >= 95%。

#### P1-C 趋势与口语时窗
- 改造项：
  - `trend/colloquial` 场景必须返回时间窗口（日期或明确时段）。
  - 所有窗口输出统一锚定时区（默认 `Asia/Shanghai`，可配置）。
  - 一旦输出“具体日期”，必须自动校验“日期-星期”一致性。
  - 相对时窗词（如“这周/近几天/这两天”）统一映射到显式日期区间。
- 测试样例：
  - 输入：`给我算算本周运势`
  - 输入：`近哪几天气场更顺`
  - 输入：`这几天哪天更顺一点？`
- 通过标准：
  - 趋势窗口命中率 >= 90%。
  - 口语时窗命中率 >= 90%。
  - 日期-星期一致率 >= 99%。

#### P1-D 时序一致性护栏
- 改造项：
  - 新增 `temporal_guard`：校验回答中的日期、星期、相对时窗是否与系统当前时间一致。
  - 新增 `date_window_resolver`：对“这周/下周/最近几天”先解析窗口，再生成文案。
  - 新增 `time_anchor` 注入：在结构对象内显式写入 `now_ts`、`tz`、`window_start`、`window_end`。
- 测试样例：
  - 输入：`这周整体节奏怎么排更稳？`（应输出与当周一致的窗口）
  - 输入：`最近几天我怎么安排会更顺一点？`（应输出可校验日期段）
  - 输入：`现在是几月几号星期几？`（保持 time 分支正确）
- 通过标准：
  - `temporal_consistency_hit_rate >= 99%`（覆盖 time 与非 time 分支）。
  - 出现日期时 `weekday_mismatch_count = 0`。

### P2：去模板化与稳定性

#### 改造项
- `QA 渲染器`：结构化输出，供回归与门禁。
- `用户渲染器`：基于结构对象生成自然表达，不固定六段。
- 增加会话去重：近 N 轮相似模板检测与改写。
- 建议从“模板表优先”改为“证据点驱动优先”。
- 增加可观测性：记录 `strategy_id`、`fallback_path`、`template_score`、`rewrite_trigger`。

#### 测试样例
- 用例 P2-01：连续追问
  - 序列：`本周运势如何` -> `我这周先做什么` -> `最近怎么安排更顺`
  - 预期：三轮有效差异，且都贴题。
- 用例 P2-02：同问题多次问
  - 输入：`给我算算本周运势`（同会话重复 3 次）
  - 预期：结构一致但措辞和分析侧重点不完全相同。

#### 通过标准
- `template_signature_rate <= 20%`
- `session_repeat_rate <= 15%`
- 连续追问最大相似度 `< 0.85`
- 不稳定分支复现率 >= 95%（同输入 + 同参数可复放）。

### P3：质量门禁 + 灰度发布 + 自动回滚

#### 改造项
- CI 门禁指标：
  - `direct_answer_hit_rate`
  - `clarify_hit_rate`
  - `trend_window_hit_rate`
  - `colloquial_window_hit_rate`
  - `temporal_consistency_hit_rate`
  - `template_signature_rate`
  - `persona_consistency`
  - `fallback_rate`
  - `observability_coverage`
  - `p95_latency_delta`
- 灰度策略：10% -> 50% -> 100%。
- 任一核心指标破线自动回滚。

#### 测试样例
- 用例 P3-01：44 条门禁集（24 条核心回归 + 20 条补充集）全部跑通。
- 用例 P3-02：模板重复率超阈值，触发发布阻断。
- 用例 P3-03：时延劣化超阈值，触发自动降级。
- 用例 P3-04：日期/星期错配，触发发布阻断。

#### 通过标准（建议阈值）
- `direct_answer_hit_rate >= 95%`
- `clarify_hit_rate >= 95%`
- `trend_window_hit_rate >= 90%`
- `colloquial_window_hit_rate >= 90%`
- `temporal_consistency_hit_rate >= 99%`
- `template_signature_rate <= 20%`
- `persona_consistency >= 90%`
- `fallback_rate <= 3%`
- `observability_coverage >= 99%`
- `p95_latency_delta <= +150ms`

## 9. 风险与控制
- 风险 1：一次性改动过大导致线上波动。
  - 控制：分阶段 + 开关化 + 灰度。
- 风险 2：自然表达增强后可测性下降。
  - 控制：保留 QA 渲染器与结构对象，门禁基于结构判定。
- 风险 3：吉伊口吻与严谨性冲突。
  - 控制：设人格规则白名单与红线，门禁纳入人设一致性评分。
- 风险 4：同维度结果随机波动。
  - 控制：固定测试集、固定温度与提示版本，加入稳定性指标。
- 风险 5：日期/星期错配导致用户信任损失（即使主结论正确）。
  - 控制：`temporal_guard` 前置 + 门禁阻断 + 灰度监控告警。
- 风险 6：指标定义模糊导致 CI 误判。
  - 控制：补齐指标口径文档，门禁脚本与文档同版本管理。

## 10. 实施顺序建议（最终）
1. 先做 P0（上线安全网）。
2. 再做 P1（先修“答到点上”与时序一致性）。
3. 再做 P2（去模板化与稳定性）。
4. 最后做 P3（门禁与灰度固化）。

## 11. 当前结论
- 当前“运势分析不够”的核心不是模型能力不足，而是系统把不同问题压成了同构输出。
- V2 的关键不是先换话术，而是先补“问题类型识别 + 澄清 + 时间窗口 + 决策直答”。
- 在 docs/13 的实测证据下，V2 必须以“先答到点上 + 先保证时序一致，再做风格优化”为执行顺序。

## 12. 指标口径与测试集契约（V2.1 新增）

### 12.1 指标口径（用于 CI 与灰度统一判定）

| 指标 | 定义 | 统计口径 |
|---|---|---|
| `direct_answer_hit_rate` | 决策/比较类问题首句出现明确二选一或排序结论 | 命中条数 / 决策类总条数 |
| `clarify_hit_rate` | 缺关键信息问题先澄清再解读 | 命中条数 / 澄清类总条数 |
| `trend_window_hit_rate` | trend 问题给出可解析时间窗口 | 命中条数 / trend 总条数 |
| `colloquial_window_hit_rate` | 口语时窗问题给出显式日期段或明确时段 | 命中条数 / colloquial 总条数 |
| `temporal_consistency_hit_rate` | 回答中的日期、星期、相对时窗与 `now_ts + tz` 一致 | 命中条数 / 含时间表达总条数 |
| `template_signature_rate` | 命中固定模板签名字段阈值的回答占比 | 模板化条数 / 总条数 |
| `session_repeat_rate` | 同会话内高相似回答占比 | 高相似 pair 数 / 会话 pair 总数 |
| `observability_coverage` | 关键日志字段齐全的请求占比 | 日志完整条数 / 总请求条数 |
| `persona_consistency` | 人设规则白名单命中且未触发红线 | 通过条数 / 总条数 |

### 12.2 测试集契约（避免“样本不足导致阈值失真”）

- 门禁集总量固定为 44 条：
  - 核心回归 24 条（decision/clarify/trend/colloquial/sequence/time 各 4 条）。
  - 补充复测 20 条（来自历史失败模式回放）。
- 统计与发布约束：
  - 任一核心维度样本数 < 4，视为门禁无效。
  - 每次发版固定 `prompt_version`、`temperature`、`tz`，否则结果不入库。
  - 门禁失败必须输出 `reason_code` Top3，禁止仅返回“未通过”。
