# 运势分析 V2 完整解决方案清单

> 基于：
> - `docs/12-运势分析深度问题与V2修订方案.md`
> - `docs/13-运势分析V2问题测试结果-2026-02-27.md`

## 1. 目标与范围

- [ ] 以“先答到点上，再去模板化，最后门禁固化”为执行顺序推进 V2。
- [ ] 在不牺牲现有可用性的前提下，修复决策直答、星座澄清、时窗命中、时序一致性与同质化问题。
- [ ] 保持现有 time 分支正确性，不产生回归退化。

## 2. P0 开关化与回滚能力（上线安全网）

### 2.1 Feature Flag 与优先级

- [ ] 新增开关：`intent_v2`。
- [ ] 新增开关：`clarify_v2`。
- [ ] 新增开关：`window_v2`。
- [ ] 新增开关：`render_v2`。
- [ ] 新增开关：`quality_gate_v2`。
- [ ] 开关优先级落地：`intent_v2 -> clarify_v2/window_v2 -> render_v2 -> quality_gate_v2`。
- [ ] 开关切换后 1 分钟内生效。
- [ ] 回滚不依赖重新发版。

### 2.2 非法组合与降级策略

- [ ] 实现非法组合约束：`quality_gate_v2=on + render_v2=off` 时仅允许基于结构对象判定质量。
- [ ] 实现非法组合约束：`window_v2=on + intent_v2=off` 时强制降级旧链路。
- [ ] 降级时记录 `reason_code=window_without_intent`。

### 2.3 可观测性基础字段

- [ ] 关键日志字段统一输出：`route_path`。
- [ ] 关键日志字段统一输出：`reason_code`。
- [ ] 关键日志字段统一输出：`flag_snapshot`。
- [ ] 关键日志字段统一输出：`domain_intent`。
- [ ] 关键日志字段统一输出：`question_type`。
- [ ] 关键链路日志覆盖率达到 `>=99%`。

## 3. P1 先修“答到点上”

### 3.1 P1-A 决策型首句直答

- [ ] 建立 `question_type=decision/comparison` 识别规则。
- [ ] 决策/比较类输出首句必须包含明确结论（二选一或排序）。
- [ ] 对“开源 vs 守财”“扩收入 vs 控支出”问法建立专项断言。
- [ ] 指标达标：`direct_answer_hit_rate >= 95%`。

### 3.2 P1-B 星座缺信息先澄清

- [ ] 对“星座运势”且未给星座名的问题优先输出澄清提问。
- [ ] 澄清命中时禁止落入八字链路输出。
- [ ] 澄清后再进入星座链路进行正式解读。
- [ ] 指标达标：`clarify_hit_rate >= 95%`。

### 3.3 P1-C 趋势与口语时窗命中

- [ ] trend 场景强制返回时间窗口（日期段或明确时段）。
- [ ] colloquial 场景强制返回显式窗口（如“近几天/哪几天”对应日期段）。
- [ ] 统一时区锚点（默认 `Asia/Shanghai`，可配置）。
- [ ] 相对时间词映射为显式窗口（这周/下周/近几天/这两天）。
- [ ] 指标达标：`trend_window_hit_rate >= 90%`。
- [ ] 指标达标：`colloquial_window_hit_rate >= 90%`。

### 3.4 P1-D 时序一致性护栏

- [ ] 新增 `date_window_resolver`，先解析窗口再生成文案。
- [ ] 新增 `temporal_guard`，校验日期、星期、相对时窗一致性。
- [ ] 在结构对象写入 `now_ts`。
- [ ] 在结构对象写入 `tz`。
- [ ] 在结构对象写入 `window_start`。
- [ ] 在结构对象写入 `window_end`。
- [ ] 指标达标：`temporal_consistency_hit_rate >= 99%`。
- [ ] 指标达标：`weekday_mismatch_count = 0`。

## 4. P2 去模板化与稳定性

### 4.1 渲染层分离

- [ ] 建立 `QA 渲染器`（结构稳定、可回归判定）。
- [ ] 建立 `用户渲染器`（自然表达、不固定六段模板）。
- [ ] 按 `question_type` 选择渲染骨架，避免同题同构。

### 4.2 证据驱动与重复抑制

- [ ] 建议生成从“topic+strength 固定表”升级为“证据点优先、模板兜底”。
- [ ] 引入同会话近 N 轮相似度检测与改写触发。
- [ ] 记录 `strategy_id`。
- [ ] 记录 `fallback_path`。
- [ ] 记录 `template_score`。
- [ ] 记录 `rewrite_trigger`。

### 4.3 稳定性指标

- [ ] 指标达标：`template_signature_rate <= 20%`。
- [ ] 指标达标：`session_repeat_rate <= 15%`。
- [ ] 指标达标：连续追问最大相似度 `< 0.85`。
- [ ] 指标达标：同输入 + 同参数分支复现率 `>= 95%`。

## 5. P3 质量门禁、灰度、自动回滚

### 5.1 门禁指标落地

- [ ] 门禁引入 `direct_answer_hit_rate`。
- [ ] 门禁引入 `clarify_hit_rate`。
- [ ] 门禁引入 `trend_window_hit_rate`。
- [ ] 门禁引入 `colloquial_window_hit_rate`。
- [ ] 门禁引入 `temporal_consistency_hit_rate`。
- [ ] 门禁引入 `template_signature_rate`。
- [ ] 门禁引入 `session_repeat_rate`。
- [ ] 门禁引入 `persona_consistency`。
- [ ] 门禁引入 `fallback_rate`。
- [ ] 门禁引入 `observability_coverage`。
- [ ] 门禁引入 `p95_latency_delta`。

### 5.2 测试集契约与脚本升级

- [ ] 固定门禁集总量为 44 条（24 核心回归 + 20 补充复测）。
- [ ] 每个核心维度样本数不得少于 4 条。
- [ ] 固定 `prompt_version`、`temperature`、`tz` 才允许结果入库。
- [ ] 门禁失败输出 `reason_code Top3`，禁止仅输出“未通过”。

### 5.3 灰度与回滚

- [ ] 灰度发布阶段：10% -> 50% -> 100%。
- [ ] 任一核心指标破线自动回滚到旧链路。
- [ ] 回滚后保留故障窗口日志与指标快照，便于复盘。

## 6. 验收阈值（最终 DoD）

- [ ] `direct_answer_hit_rate >= 95%`
- [ ] `clarify_hit_rate >= 95%`
- [ ] `trend_window_hit_rate >= 90%`
- [ ] `colloquial_window_hit_rate >= 90%`
- [ ] `temporal_consistency_hit_rate >= 99%`
- [ ] `weekday_mismatch_count = 0`
- [ ] `template_signature_rate <= 20%`
- [ ] `session_repeat_rate <= 15%`
- [ ] 连续追问最大相似度 `< 0.85`
- [ ] `persona_consistency >= 90%`
- [ ] `fallback_rate <= 3%`
- [ ] `observability_coverage >= 99%`
- [ ] `p95_latency_delta <= +150ms`

## 7. 推荐执行顺序

- [ ] 第 1 步：完成 P0（先有安全网）。
- [ ] 第 2 步：完成 P1（先保证答对题和时序正确）。
- [ ] 第 3 步：完成 P2（再解决模板同质化与稳定性）。
- [ ] 第 4 步：完成 P3（门禁固化、灰度发布、自动回滚）。
