# 运势/命理/占卜回复问题分析与优化方案

更新时间：2026-02-26  
项目路径：`/Users/yayauu/PycharmProjects/fortune-telling`

## 1. 问题结论（摘要）

当前“回复泛泛而谈、空理论、鸡汤化”的根因不是单点，而是以下链路叠加：

1. 输入到模型前被多层“风格指令+资料提示”拼接，污染用户真实问题语义。
2. 命理问题未稳定触发八字工具，很多轮次靠模型自由发挥。
3. 八字工具返回字段太薄，模型缺少足够命理依据可引用。
4. 系统提示词约束过多且相互竞争，模型偏向安全模板回答。
5. 温度参数过低（`temperature=0`），加重措辞模式化。
6. 前端追问建议语料偏情绪疗愈，持续把对话拉向“泛情绪建议”。

---

## 2. 现象与证据

1. 命理问答出现“有情绪、有安慰、少依据”的回复。
2. 同类问题多轮回答句式相近，“先给结论+再给行动建议”结构重复。
3. 有些轮次会出现命理术语，但缺少可核对的结构化依据（如五行分布、喜用神、流年关联）。

关键代码证据：

1. 输入拼接与风格注入：`build_profile_augmented_query`、`build_style_instruction`、`/chat` 拼接后再送 Agent。  
文件：`server.py`（约第 707、774、1527、1537 行）
2. 系统提示词规则过多：角色设定、口头禅、工具限制、惩罚式规则并存。  
文件：`server.py`（约第 470-529 行）
3. 工具调用仅“优先”而非“强制路由”：  
文件：`server.py`（约第 518 行）
4. 八字工具返回字段较少，常退化为短文本：  
文件：`mytools.py`（约第 81-100 行）
5. 模型温度固定 0：  
文件：`models.py`（约第 16-25 行）

文件：`static/js/main.js`（约第 20-133 行）

## 3. 根因拆解（一步步）

1. 语义入口被“包裹”  
后端将资料提示与表达约束直接拼进用户问题正文，导致模型优先遵循格式口令，而不是先理解问题本体。

2. 历史记忆被污染  
这些拼接文本进入对话记忆后，会在后续轮次继续影响输出风格，造成模板痕迹累积。

3. 命理数据供给不足  
未调用八字工具时，模型只能生成“看起来合理”的泛建议；调用工具时若工具结果过薄，仍会二次泛化。

4. 规则冲突导致“保守生成”  
“要可爱”“要准确”“要不模板”“要先结论”“工具限制”同时存在，模型会走最稳妥的通用表达。

5. 低温度放大重复  
温度为 0 时，模型更容易重复固定句法和固定论证路径。

---

## 4. 优化方案（按优先级）

## P0：先解决“依据不足”

1. 建立命理请求路由器（后端显式路由，不靠 Agent 自发）
- 规则：命理关键词命中且资料满足（姓名+生日，时辰可选）时，先执行 `bazi_cesuan`。
- 规则：资料不足时仅追问关键缺口，不输出完整结论。

2. 升级 `bazi_cesuan` 输出为结构化高信息密度
- 输出建议：`bazi`、`day_master`、`wuxing_scores`、`xiyongshen`、`major_luck`、`year_hint`、`confidence`。
- 增加：超时、重试、错误码与降级路径，避免工具失败直接回泛话术。

## P1：再解决“模板化”

1. 重构提示词注入方式
- 用户原问保持原样，不再把风格与资料硬拼到 `query`。
- 风格规则和资料信息通过独立 system/developer message 传入，不写入 chat memory。

2. 精简系统提示词
- 删除惩罚式措辞和互相冲突条款。
- 只保留核心三条：吉伊口吻、先结论后依据、不编造事实。

3. 调整生成参数
- 将温度从 `0` 调整到 `0.3-0.5`（建议 `0.4`）。

## P2：最后做“稳定性工程”

1. 建立命理专项回归集（20-30 条）
- 场景覆盖：桃花、事业、财运、流年、占卜、资料缺失追问。
- 每次改动后自动跑回归，输出通过率。

2. 定义质量指标
- 工具调用率（命理问题应达标）
- 依据密度（是否含可核对命理字段）
- 空话率（泛化句占比）
- 口吻自然度（人工抽检）

3. 前端建议语料分流
- 命理会话优先推荐“命理细问”按钮，而不是持续推荐情绪疗愈问句。

## 5. 建议实施顺序（执行清单）

1. 第 1 步：实现命理路由器与工具强制前置（P0）。
2. 第 2 步：扩充 `bazi_cesuan` 结构化返回（P0）。
3. 第 3 步：重构 prompt 注入，清理 query 拼接（P1）。
4. 第 4 步：精简系统提示词 + 调温度（P1）。
5. 第 5 步：补命理回归集与评测脚本（P2）。
6. 第 6 步：前端建议库按场景分流（P2）。

## 6. 风险与注意事项

1. 过度放开风格约束会造成“人设漂移”，需用最小必要规则守住吉伊口吻。
2. 过度强调术语会让体验变硬，需保持“结论-依据-行动”三段平衡。
3. 外部八字接口稳定性会直接影响回答质量，必须有降级策略。
4. 回归集需包含“真实用户脏输入”，否则上线后仍可能退化。

---

## 7. 预期效果

1. 运势/命理/占卜问题回复更具体，依据可核对，空话率下降。
2. 吉伊口吻保留但不过度卖萌，人设更自然。
3. 同类问题答复多样性提升，模板感明显降低。
4. 出问题时可通过指标快速定位是“路由问题、工具问题还是提示词问题”。

## 8. 最终解决方案（补充版）

在“强路由 + 结构化工具输出 + 分流后处理 + 质量门禁”主框架上，补充以下执行定义，保证可落地、可回归、可观测。

### 8.1 命理强路由（唯一入口与优先级）

1. 在 `/chat` 建立唯一命理入口函数（如 `route_fortune_pipeline(query, profile)`）。
2. 优先级固定：
   1. 命理意图命中且资料齐全 -> 直接走命理专用链；
   2. 命理意图命中但资料不足 -> 仅追问缺失项（姓名/生日；时辰可选）；
   3. 其余 -> 走通用 Agent。
3. 命理链结果返回后，不再回落到通用链重写，避免被泛化模板覆盖。

### 8.2 命理输出 JSON 合约（强类型）

`bazi_cesuan` 统一返回结构化 JSON，并以 Pydantic 校验。建议字段：

- `topic`: `daily|love|wealth|career|study`
- `bazi`: string
- `day_master`: string
- `strength`: `strong|weak|balanced`
- `xiyongshen`: string
- `jishen`: string
- `wuxing_scores`: object（`metal/wood/water/fire/earth`）
- `fortune_signals`: object（如 `love/wealth/career` 方向短评）
- `advice`: array[string]（1-3 条可执行动作）
- `confidence`: float（0-1）
- `source`: `yuanfenju`
- `error`: optional object（失败时）

要求：

1. 字段缺失时必须给默认值，不允许返回自由散文替代。
2. 渲染层只消费 JSON，不直接消费工具原始长文本。

### 8.3 外部 API 失败降级策略（可追踪）

1. 超时与重试：
   - 单次超时：`3s`
   - 最大重试：`2` 次（指数退避）
2. 失败路径：
   - 返回“轻量命理保底模板 + 一条明确下一动作”
   - 不回退为纯情绪安抚文
3. 错误码规范：
   - `FORTUNE_TIMEOUT`
   - `FORTUNE_UPSTREAM_5XX`
   - `FORTUNE_PARSE_FAILED`
4. 记录日志字段：
   - `session_id`、`user_uuid`、`query_type`、`error_code`、`latency_ms`

### 8.4 两阶段生成参数（避免全链路同温度）

1. 参数抽取/结构化阶段（严谨）：
   - `temperature = 0 ~ 0.2`（建议 `0.1`）
2. 文案渲染阶段（自然）：
   - `temperature = 0.3 ~ 0.5`（建议 `0.4`）
3. 不建议全链路统一 `0.4`，否则参数抽取稳定性下降。

### 8.5 分流后处理规则（命理 vs 通用）

1. 命理后处理：`sanitize_fortune_output`
   - 仅做：去个人信息回显、去重复、长度裁剪、术语密度下限检查
   - 禁止：将命理依据替换成固定“鸡汤模板句”
2. 通用后处理：`sanitize_general_output`
   - 保留现有语气与可读性优化逻辑

### 8.6 可观测性与质量门禁

新增埋点指标：

1. `fortune_route_hit_rate`（命理路由命中率）
2. `fortune_tool_success_rate`（八字工具成功率）
3. `fortune_field_completeness_rate`（结构字段完整率）
4. `profile_echo_violation_rate`（个人信息回显违规率）
5. `template_repeat_rate`（模板重复率）

发布门禁（未达标不发布）：

1. 命理问题工具命中率 >= 90%
2. 每条回复可核对命理字段 >= 2
3. 个人信息原文回显违规率 = 0
4. 模板重复率 < 20%
5. 10 条样本中 >= 8 条“有依据且不空泛”

### 8.7 回归集与测试补充

在 `docs/05-测试用例.md` 增加“命理质量回归组”：

1. 今日运势/桃花/财运/事业运/流年共 20-30 条样本
2. 每条包含：
   - 输入
   - 期望命中路由
   - 期望 JSON 字段最小集
   - 个人信息回显断言
   - 文本重复率断言
3. CI 中增加命理回归脚本，结果输出通过率与失败样例。
