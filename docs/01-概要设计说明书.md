# 概要设计说明书

## 1. 文档目的
本文档描述“吉伊大师命理咨询系统”当前已实现版本的总体设计，作为开发、联调、测试与运维基线。

## 2. 系统概述
系统是一个基于 FastAPI 的命理问答 Web 应用，核心能力包括：
- 手机号验证码注册/登录（演示版）
- 基于 Cookie 的登录态鉴权
- 聊天问答（含快捷回复 + Agent 工具调用）
- 用户资料抽取与持久化（姓名/出生日期/出生时刻）
- 网页知识入库到本地 Qdrant

## 3. 总体架构
系统采用单体后端 + 模板前端 + 外部依赖服务架构。

- 表现层：`templates/*.html` + `static/js/*.js` + `static/css/*.css`
- 接口层：FastAPI 路由（`server.py`）
- 业务层：鉴权、资料抽取、聊天编排、输出清洗
- AI 层：LangChain Agent（`Master`）+ 工具函数（`mytools.py`）
- 存储层：
  - MySQL：用户主数据、用户资料、登录会话审计
  - Redis：验证码、验证码冷却、登录态、短期聊天历史
  - Qdrant（本地）：知识向量库
- 外部服务：阿里百炼模型、SerpAPI、缘分居 API

## 4. 模块划分
- 应用入口与路由：`server.py`
- 模型客户端工厂：`models.py`
- 工具函数：`mytools.py`
- 环境配置：`config.py`
- 日志：`logger.py`
- 前端页面：`templates/index.html`、`login.html`、`register.html`、`forgot_password.html`
- 前端脚本：`static/js/main.js`、`static/js/login_multi.js`、`static/js/auth.js`、`static/js/reset_password.js`

## 5. 核心业务流程

### 5.1 注册/登录（验证码）
1. 前端调用 `POST /auth/send_code`（按 `scene=login/register`）。
2. 后端校验手机号与冷却时间，验证码写入 Redis（TTL=300s）。
3. 前端调用 `POST /auth/verify`（`mode=login/register`）。
4. 注册时自动创建用户：`uuid`、`account=JIYI-XXXXXXXX`，落 MySQL。
5. 后端写 Redis 登录态 + MySQL 会话审计，设置 `jiyi_auth_token` Cookie。

### 5.2 聊天
1. 校验登录态（Redis Session）。
2. 通过手机号查 MySQL 用户，`session_id` 绑定用户 `uuid`。
3. 读取 Redis 历史消息，抽取资料并补齐到 MySQL `user_profile`。
4. 命中命理/占卜意图时优先走命理专用路由（资料不足先追问，资料齐全则调用命理工具并结构化渲染）。
5. 命中快捷回复（时间/问候）直接返回。
6. 其余问题进入 Agent + tools，输出清洗后返回。

### 5.3 知识入库
1. 调用 `POST /add_urls?URL=...`。
2. 抓取网页并切分文本。
3. 向量化后写入本地 Qdrant 集合。

## 6. 部署架构
- 本地运行：`python server.py`（默认 8000）
- Docker Compose 运行：
  - `numerology`（应用）
  - `redis`（会话/验证码/聊天历史）
  - `mysql`（用户与审计数据）

## 7. 非功能设计
- 可维护性：配置集中在 `.env` + `config.py`
- 可扩展性：工具函数通过 `@tool` 注册，易于扩展
- 可观测性：`loguru` 打印关键链路日志
- 质量观测：提供 `GET /quality/metrics` 聚合命理路由命中率/工具成功率/字段完整率等指标
- 安全性（当前版本）：验证码会回传 `debug_code`，仅适合开发/演示环境

## 8. 当前限制
- 验证码仍以 `debug_code` 明文回传（仅适合开发/演示，生产应关闭）。
- Cookie 未启用 `secure=True`（HTTPS 生产环境建议开启）。
- 外部依赖失败时（模型/API）会出现能力降级。
- `requirements.txt` 为 UTF-16 编码，构建流程需转换处理。
