# 数据库设计说明书

## 1. 说明
当前项目数据层为 MySQL + Redis + Qdrant：
- MySQL：用户主数据、用户资料、会话审计
- Redis：验证码、验证码冷却、登录态、聊天历史
- Qdrant：本地知识向量库

## 2. MySQL 设计（已实现依赖）

## 2.1 `users`
用途：用户主数据。

主要字段：
- `id`（PK）
- `uuid`（唯一）
- `account`（唯一，格式 `JIYI-XXXXXXXX`）
- `phone`（唯一）
- `password_hash`（当前短信注册用户占位值）
- `status`
- `created_at`、`updated_at`

代码使用：
- 查询：`_get_user_by_phone`、`_get_user_by_uuid`
- 新建：`_create_user_by_phone`

## 2.2 `user_profile`
用途：用户资料。

主要字段：
- `user_id`（PK/FK -> users.id）
- `name`
- `birth_date`
- `birth_time`
- `updated_at`

代码使用：
- 查询：`_get_profile_by_user_id`
- 更新：`_merge_profile_to_db`

更新规则：仅补齐缺失资料，不覆盖已有值。

## 2.3 `auth_sessions`
用途：登录会话审计。

主要字段：
- `id`（PK）
- `user_id`
- `token_hash`（SHA-256）
- `login_type`（当前使用 `sms`）
- `device_info`
- `ip`
- `expires_at`
- `created_at`
- `revoked_at`

代码使用：
- 登录写入：`_save_auth_session_to_db`
- 登出撤销：`_revoke_auth_session_in_db`

## 3. Redis 设计（已实现）

## 3.1 登录态
- Key：`auth:session:{token}`
- Value：JSON（`phone`、`user_uuid`）
- TTL：30 天
- 读写函数：`_set_auth_session`、`_get_auth_session`、`_delete_auth_session`

## 3.2 验证码
- Key：`auth:sms:{scene}:{phone}`
- Value：6 位验证码
- TTL：300 秒
- 读写函数：`_set_sms_code`、`_get_sms_code`、`_delete_sms_code`

## 3.3 验证码冷却
- Key：`auth:sms:cooldown:{scene}:{phone}`
- Value：`1`
- TTL：60 秒
- 查询函数：`_sms_cooldown_ttl`

## 3.4 聊天历史
- 组件：`RedisChatMessageHistory`
- 会话 ID：用户 `uuid`
- TTL：`SESSION_TTL_SECONDS`（默认 600 秒）
- 用途：给 Agent 提供短期上下文

## 4. Qdrant 设计
- 路径：`VECTOR_DB_PATH`（默认 `./local_qdrand`）
- 集合：`VECTOR_COLLECTION_NAME`（默认 `local_documents`）
- 写入入口：`POST /add_urls`
- 检索入口：`get_info_from_local_db`

## 5. 逻辑关系
- `users (1) -> (1) user_profile`
- `users (1) -> (N) auth_sessions`
- `users.uuid (1) -> (N) RedisChatMessageHistory`
- URL 文档切片 `(N) -> (1) Qdrant collection`

## 6. 一致性策略
- 用户与资料以 MySQL 为准。
- 会话在线态与验证码以 Redis 为准。
- 聊天历史为 Redis 短期存储，不落 MySQL（当前版本）。

## 7. 风险与注意事项
- MySQL 建表不完整会导致登录/聊天资料流程失败。
- Redis 清空后，登录态和验证码会失效（需重新登录/发送验证码）。
- 生产环境应移除 `debug_code` 明文返回。
